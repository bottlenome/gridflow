# gridflow アーキテクチャドキュメント作成計画

## 目的

本計画は、gridflow のアーキテクチャドキュメントを**アーキテクチャ中心設計手法（Architecture Centric Design Method: ACDM）**に基づいて作成するための進め方を定義する。

ACDM では、アーキテクチャドライバー（ビジネス目標・品質属性・制約・主要機能）がアーキテクチャ設計を駆動し、複数のビューを反復的に行き来しながら精緻化していく。本計画もその原則に従い、**シーケンシャルな工程ではなく、反復的な分析・設計サイクル**として進める。

## 反復的な作業の進め方

ACDM に従い、以下のように反復しながら進める。各ラウンドで複数のセクションに手を入れ、ビュー間の整合性を保つ。

### Round 0: コンセプトからのアーキテクチャ重要事項の転記

**目的:** 計画書に既に存在するアーキテクチャ上の重要事項を構造化して転記し、以降のラウンドの入力とする

**作業内容:**
- はじめに（セクション 1）を作成
- ビジネス目標の整理（セクション 2.1）
  - 計画書セクション 0「目的」、1.2「なぜ今これをやるのか」、2.1「コア価値」から抽出
- アーキテクチャドライバーの転記・分類（セクション 2.2）
  - 主要機能要求: 計画書セクション 3.1 の P0 機能（A〜F）を要求として再構成
  - 品質属性シナリオ: 計画書セクション 10.1 の KPI を ACDM の形式（刺激源・刺激・成果物・環境・応答・応答測定値）に変換
  - 制約: 計画書セクション 1.3（1人+AI開発）、5.1（Linux コンテナ）、5（Python 統一）を制約として整理
  - アーキテクチャ関心事: 計画書セクション 8（OSS 分岐判断）、1.2（既存ツールの課題）を関心事として整理

**なぜ Round 0 か:**
計画書にはアーキテクチャ上の重要な判断材料が散在している。これをまず ACDM のフレームワークで構造化しておくことで、Round 1 以降のユースケース分析や設計判断の土台ができる。新たな分析ではなく、既存情報の再編成が主作業である。

**成果物:** `docs/architecture/01_introduction.md`, `docs/architecture/02_architecture_significance.md`（2.1〜2.2 の初版）

---

### Round 1: ユースケース分析 + ドライバー精緻化

**目的:** システムが「何をするか」を明確にし、Round 0 で転記したドライバーをユースケースの観点から精緻化する

**作業内容:**
- 計画書からユースケースを抽出し、ユースケース図を作成（セクション 4.1）
  - 全 10 ユースケース: シミュレーション実行 / Scenario Pack 管理 / ベンチマーク評価 / 起動・終了 / ログ / デバッグ / インストール・セットアップ / アップデート・アンインストール / 結果参照・データエクスポート / LLM による実験指示
  - アクター識別: 研究者（L1-L4）、CI/CD システム、LLM Agent（学生・教員は L1-L4 に包含。AC-3 参照）
- ユースケースシナリオ UC-01〜UC-10 をテキストで記述（セクション 4.2）
  - 各シナリオ: 事前条件・トリガー・基本フロー・代替フロー・例外フロー・事後条件
- Round 0 のドライバーをユースケースの観点から見直し・補完（セクション 2.2 更新）

**Round 0 との関係:**
Round 0 で転記した機能要求を、ユースケースとして具体化する。ユースケース分析の過程で、見落としていたドライバーや品質属性が見つかれば 2.2 に追加する。

**成果物:** `docs/architecture/04_dynamic_view.md`（4.1〜4.2 の初版）、`docs/architecture/02_architecture_significance.md`（更新）

---

### Round 2: 静的ビュー初版 + アーキテクチャ戦略

**目的:** ユースケースを実現するための構造を設計する

**作業内容:**
- ブロック図: ユースケースを実現するサブシステム分割（セクション 3.1）
- クラス図: 各サブシステムの主要クラス・インターフェース（セクション 3.2）
  - 特に Connector の抽象化と L1-L4 の拡張ポイント
- アーキテクチャ戦略: ドライバーに応じた高レベル方針（セクション 2.3）
- アーキテクチャパターン・戦術: 品質属性を実現する具体手法（セクション 2.4）

**Round 1 との関係:**
ユースケースとドライバーが「何を実現するか」を定義した上で、「どう実現するか」を構造として表現する。

**成果物:** `docs/architecture/03_static_view.md`（3.1〜3.2 の初版）、`docs/architecture/02_architecture_significance.md`（2.3〜2.4 追加）

---

### Round 3: 動的ビュー深化 + 配置図

**目的:** 静的構造の上で動的な振る舞いを検証し、物理配置を決定する

**作業内容:**
- シーケンス図 UC-01〜UC-10（UC-08 を除く）: 主要ユースケースの実行フローを静的ビューの要素（クラス・コンポーネント）を使って記述（セクション 4.3）
  - このとき静的ビューの不足（足りないクラス、不明なインターフェース）が見つかれば Round 2 の成果物を更新
- 配置図: Docker コンテナ構成・ホスト環境（セクション 3.3）
  - シーケンス図で明らかになるプロセス間通信・コンテナ間通信を反映

**成果物:** `docs/architecture/04_dynamic_view.md`（4.3 追加）、`docs/architecture/03_static_view.md`（3.3 追加 + 必要に応じ 3.1〜3.2 更新）

---

### Round 4: ビュー間対応 + 評価 + ADR

**目的:** 全ビューの整合性を確認し、アーキテクチャを評価する

**作業内容:**
- ビュー間の対応表を作成（セクション 5）
  - 静的要素 ↔ 動的要素の対応
  - ユースケース ↔ コンポーネントのトレーサビリティ
  - ドライバー ↔ ビュー要素のトレーサビリティ
- アーキテクチャ評価: 品質属性シナリオに対する設計の妥当性を検証（セクション 6）
  - リスク・感度点・トレードオフの特定
- 設計判断の記録 ADR-001〜ADR-007（セクション 7）
- 全セクションの横断的な整合性確認・用語統一
- 付録の完成（セクション A〜D）

**成果物:** `docs/architecture/05_view_mapping.md`, `docs/architecture/06_architecture_evaluation.md`, `docs/architecture/07_adr.md`, `docs/architecture/08_appendix.md`, `docs/architecture/README.md`

## 各ラウンドの進め方

- 各ラウンドの成果物を提示し、レビューを受けてから次のラウンドに進む
- ただし、後のラウンドで前のラウンドの成果物に修正が必要になることは**想定内**であり、随時更新する
- 図は全て **Mermaid 記法** で記述し、GitHub 上でレンダリング可能にする
- 言語は **日本語** とする（コード中のクラス名・技術用語は英語）

---

## 留意事項

- 現時点では実装コードが存在しないため、クラス図・シーケンス図は計画書の設計意図に基づく**設計時アーキテクチャ**（notional architecture）として記述する
- 実装が進んだ段階で、実装との差異を反映する改訂を想定する
- ACDM の原則に従い、**アーキテクチャドライバーが設計判断を駆動する**構造を維持する
- 各ビューは独立した成果物ではなく、**相互に参照・制約し合う関係**にある
