## 1. 本計画の目的

アーキテクチャドキュメント（ACDM）を入力として、IPA（独立行政法人情報処理推進機構）のソフトウェア開発ガイドラインに準拠した**基本設計書**を作成する。基本設計書は、アーキテクチャ上の設計判断を「実装可能な外部仕様」に落とし込むことを目的とする。

### 入力ドキュメント

| ドキュメント | パス | 役割 |
|---|---|---|
| アーキテクチャドキュメント | `docs/architecture/` (01〜08) | 設計判断の根拠、静的・動的ビュー、品質属性 |
| 計画書 | `docs/gridtwin_lab_plan.md` | プロダクト定義、機能要件、ロードマップ |
| アーキテクチャドキュメント作成計画 | `docs/architecture_document_plan.md` | 作成プロセスの定義 |

## 3. 要求一覧（ID 体系）

トレーサビリティのため、以下の ID 体系で要求を管理する。

### ID 体系

| 接頭辞 | 分類 | 例 | 出典 |
|---|---|---|---|
| `REQ-F-xxx` | 機能要求 | REQ-F-001 | 計画書 P0/P1/P2 + アーキテクチャ FR-01〜FR-07 |
| `REQ-Q-xxx` | 品質要求 | REQ-Q-001 | アーキテクチャ QA-1〜QA-11 |
| `REQ-C-xxx` | 制約 | REQ-C-001 | アーキテクチャ CON-1〜CON-5 |
| `REQ-B-xxx` | ビジネス要求 | REQ-B-001 | アーキテクチャ BG-1〜BG-4 |

### トレーサビリティマトリクス（例）

```
REQ-B-001 (BG-1) → REQ-F-001 (FR-01) → 第3章 3.2 → 第5章 5.2
                  → REQ-F-002 (FR-02) → 第3章 3.3 → 第6章 6.4
                  → REQ-Q-001 (QA-1)  → 第10章 10.1
```

## 7. 作成方針

1. **アーキテクチャドキュメントとの一貫性**: 用語・ID は ACDM ドキュメントに準拠する。基本設計書独自の用語は定義しない
2. **トレーサビリティ第一**: 全設計項目が REQ-xxx を通じてビジネス目標まで追跡可能であること
3. **段階的詳細化**: アーキテクチャドキュメントの抽象度を1段下げる。実装の詳細（クラス設計、メソッドシグネチャ等）は詳細設計書に委ねる
4. **IPA 準拠 + gridflow 最適化**: IPA の標準構成を基本とし、gridflow 固有の事情（CLI ベース、Docker 環境、研究用途）に合わせて読み替える
5. **章単位の独立作成**: 各章は独立してレビュー可能な粒度とし、Phase 順に完成させる
