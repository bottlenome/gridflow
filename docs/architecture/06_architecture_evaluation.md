# 6. アーキテクチャ評価

## 6.1 品質属性シナリオに対する評価

各 QA シナリオに対して、現在のアーキテクチャが応答測定値を達成できるか評価する。

| QA | 応答測定値 | 達成見込み | 根拠 | リスク |
|---|---|---|---|---|
| QA-1 導入容易性 | セットアップ < 30 分 | **高** | Docker Compose + Installer 自動セットアップ（UC-07）。サンプル Pack 事前登録 | Docker Desktop のインストール自体がユーザーの障壁になる可能性 |
| QA-2 初回利用効率 | TTFS < 1 時間 | **高** | セットアップ完了後、サンプル Pack で即座に `gridflow run` 可能 | サンプル Pack の品質（研究者にとって意味のある実験か）が導入体験を左右 |
| QA-3 再現性 | 2 環境以上で結果一致 | **中** | Scenario Pack に seed を含め、Docker で環境固定。ただし浮動小数点演算のアーキテクチャ間差異は外部ツール依存 | ARM64 と AMD64 で外部ツールの計算結果が微小に異なる可能性。gridflow 側では制御不能 |
| QA-4 拡張性 | L2 統合 < 100 行 | **高** | Plugin API が 1-2 メソッド（3.5）。コンストラクタ注入（M-10）。Scenario Pack の YAML 宣言的統合 | Plugin API の設計が実際の研究ニーズに合わない場合、100 行では収まらない可能性 |
| QA-5 ワークフロー効率 | ベースライン再現 < 1 時間、比較実験 < 1 日 | **中** | Scenario Pack clone → パラメータ変更 → 実行の流れ。ただし外部ツールの実行時間に依存 | 大規模系統のシミュレーションが数時間かかる場合、1 日以内は外部ツール次第 |
| QA-6 エクスポート容易性 | 変換ステップ < 3 | **高** | `gridflow results export` で 1 コマンド完結（UC-09） | エクスポート後の図表化ツール（Notebook 等）の学習コストは gridflow の責務外 |
| QA-7 ポータビリティ | AMD64 + ARM64 で全テスト通過 | **中** | Docker マルチアーキビルド + CI（M-6）。ただし外部ツールの ARM64 対応状況に依存 | GridLAB-D の ARM64 対応が不安定（FR-07 で条件付き）。pandapower は問題なし |
| QA-8 可観測性 | 全 KPI 計測可能、5 分以内の障害特定 | **高** | StructuredLogger（M-1）、エラー分類（M-2）、UC-05 の 3 モード分離 | opt-in テレメトリ（AC-6）のリテンション計測は、ユーザーが有効化しない限り取得不可 |
| QA-9 LLM 親和性 | 構造化 I/O、自己説明的エラー | **高** | JSON 出力（M-9）、resolution 必須エラー（M-2）、Ubiquitous Language（AS-1）| LLM の性能に依存する部分が大きい。gridflow 側の対応は必要条件であって十分条件ではない |
| QA-10 性能効率 | オーバーヘッド < 5% | **高** | ボトルネック分析（3.4.1）で外部ツールが支配的と確認。CDL I/O スレッド + Logger スレッドで非同期化 | 大量データの CDL 格納（ケース A）で 5% を超える可能性。ストリーミング設計で対策済み |
| QA-11 論文生産性 | 年間 10 本以上、L1 でも論文が書ける | **中** | Scenario Pack によるベースライン再現、Benchmark Harness による定量比較、L1-L4 の段階的拡張（3.5）が論文執筆を支援 | 論文の質と量はアーキテクチャだけで決まらない。研究コミュニティの受容（BG-2）と Scenario Pack の品質・多様性が前提条件 |

## 6.2 リスクと感度点

### リスク

| ID | リスク | 影響度 | 発生可能性 | 関連 QA/AC | 緩和策 |
|---|---|---|---|---|---|
| R-1 | **外部ツールの ARM64 対応が不十分** | 高 | 中 | QA-7, CON-4 | GridLAB-D を条件付き（FR-07）。pandapower は問題なし。ツール別の環境戦略で個別対応 |
| R-2 | **浮動小数点のアーキテクチャ間差異で再現性が崩れる** | 中 | 中 | QA-3 | Scenario Pack に許容誤差（tolerance）を定義可能にする。bit-exact ではなく統計的一致を検証 |
| R-3 | **Plugin API が実際の研究ニーズに合わない** | 高 | 中 | QA-4, FR-06 | 早期に複数の研究トラック（AC-4: Track 1-3）で検証。API を固める前に実研究者のフィードバックを得る |
| R-4 | **1人+AI 開発体制でアーキテクチャの複雑さを維持できない** | 高 | 低 | CON-3 | シンプルさを優先する設計判断（シングルプロセス、OSAL 不要、ミドルウェア最小化）を維持。複雑化の兆候が見えたら AC-2（段階的成長パス）に沿って判断 |
| R-5 | **Docker Desktop のライセンスコスト** | 低 | 低 | CON-2, QA-1 | 大規模組織では Docker Desktop のライセンスが必要。Podman 等の代替を Connector 実装レベルで対応可能 |

### 感度点

アーキテクチャの小さな変更が品質属性に大きく影響する箇所。

| 感度点 | 影響する QA | 説明 |
|---|---|---|
| **ConnectorInterface の設計** | QA-3, QA-4, QA-7, QA-10 | Connector I/F を変更すると全 Connector 実装と全テストに波及する。AS-4 の根幹。慎重な設計が必要 |
| **Scenario Pack のスキーマ** | QA-1, QA-3, QA-4 | スキーマ変更は全既存 Pack のマイグレーション（M-7, UC-08）を必要とする。初期設計で拡張性（metadata dict）を確保済み |
| **CDL のデータモデル** | QA-3, QA-6, QA-10 | CDL 変更はエクスポート形式と再現性に影響。CIM 互換性（AC-7）との整合も必要 |

## 6.3 トレードオフ

本アーキテクチャで意図的に選択したトレードオフを記録する。

| トレードオフ | 優先したもの | 犠牲にしたもの | 理由 |
|---|---|---|---|
| **シングルプロセス vs マルチプロセス** | シンプルさ（CON-3）、デバッグ容易性 | batch 実行の並列性能 | ボトルネックは外部ツール（3.4.1）。将来の並列 Scheduler で対応可能 |
| **ファイルシステム CDL vs DB** | 導入容易性（QA-1）、シンプルさ | クエリ性能、同時アクセス性能 | P0 ではファイルで十分。Repository パターンで将来差替え |
| **統一 Connector I/F vs ツール固有 I/F** | 拡張性（QA-4）、Connector 追加の容易さ | ツール固有の最適化 | 3.1.3 で分析。個別最適化より統一の利点が大きい |
| **JSON 内部永続化 vs バイナリ** | デバッグ容易性、LLM 親和性（QA-9） | I/O 性能 | ボトルネックは I/O ではなく外部ツール（3.4.1） |
| **OSAL なし vs OSAL あり** | シンプルさ（CON-3） | ネイティブ実行の移植性 | Docker が OS 差異を吸収（CON-2） |
| **P0 でメッセージ外部化 vs 後回し** | 将来の i18n コスト削減 | P0 の開発速度（わずかに低下） | 後から i18n を入れるコストが非常に高い（M-12） |
| **DI フレームワーク不使用 vs 使用** | シンプルさ、言語非依存 | 大規模化時の DI 管理の手間 | コンストラクタ注入で十分（M-10）。CON-3 |
