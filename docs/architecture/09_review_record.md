## 4. 指摘事項一覧

### 重要度の定義

| 重要度 | 意味 |
|---|---|
| ❌ ERROR | 整合性の破綻、情報の欠落。修正必須 |
| ⚠️ WARNING | 改善推奨。現時点では致命的ではないが、将来問題になる可能性 |
| 💡 INFO | 品質向上のための提案 |

### 指摘一覧

| ID | 重要度 | 対象 | 指摘内容 | 推奨対応 |
|---|---|---|---|---|
| REV-001 | ⚠️ | 全体 | レビュー記録が独立文書として存在していなかった。v0.4 の「デザインレビュー反映」の具体的な指摘事項と対応が追跡不能 | 本文書（09_review_record.md）の追加で対応。今後のレビューも本文書に追記する |
| REV-002 | ⚠️ | `04_dynamic_view.md` / `05_view_mapping.md` | NotebookBridge が UC-09 の注記のみで言及されており、シーケンス図上の役割が不明確 | P1 以降で Notebook 連携の UC を追加する際に、NotebookBridge のシーケンス図上の参加を明示する |
| REV-003 | 💡 | `06_architecture_evaluation.md` | QA 達成見込みが「高/中」の2段階で、定量的な根拠が「外部ツール依存」に集中している | 外部ツール依存のリスクを R-1〜R-5 で管理しているため現状は許容。P0 実装後の実測データで評価を更新することを推奨 |
| REV-004 | 💡 | `07_adr.md` | ADR のステータスが「決定済み」「保留」の2種のみ。「廃止」「置換」がない | 現時点では不要。ADR の置換が発生した際にステータス種別を拡張する |
| REV-005 | 💡 | `03b_mechanisms.md` | M-1〜M-14 の実装メカニズムが ADR-001（Python 確定）後の Python 実装方針を含むが、ADR との相互参照が一部のみ | 各メカニズムの「設計判断」欄に関連 ADR 番号を追記すると、トレーサビリティが向上する |
