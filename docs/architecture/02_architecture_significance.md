# 2. アーキテクチャの重要事項

## 2.1 ビジネス目標

計画書（セクション 0, 1.2, 2.1, 9.4, 9.5）から抽出したビジネス目標を以下に示す。

### BG-1: 電力系研究の E2E ループを高速化する

> **研究テーマではなく、研究の回し方を高速化する**（計画書 2.1）

電力系研究室が繰り返し払っている実験運用コスト（環境構築、連携コード作成、入出力整形、実験再実行、グラフ生成、表整理）を圧縮するワークフローエンジンを提供する。

### BG-2: 共同研究の再現実験基盤として普及する

> **既存論文の再現パックを一緒に作りませんか**（計画書 9.4）

初期導入先は大学研究室（国内外）および共同研究の再現実験基盤である（計画書 2.3）。導入研究室数を主要指標とし、マイルストーンとして 10 → 50 → 100 研究室を目指す（計画書 10.2）。

教育利用（授業演習・grading 等）は gridflow の直接の責務としない。研究者が自分の研究室の学生に使わせる形で自然に広がることを想定し、教員パートナーが教材化できる拡張性を確保するに留める。

### BG-3: 実験の再現性を制度として担保する

卒業と同時に実験が失われる問題を防ぎ、実験条件の再現性を Scenario Pack として制度化する（計画書 3.1 A）。

### BG-4: 研究フローと評価方法を標準化する

研究手法ではなく、研究フローと評価方法を標準化する（計画書 9.5）。共通ベンチマークによる定量比較を可能にする。

## 2.2 アーキテクチャドライバー

ビジネス目標を実現するためにアーキテクチャに影響を与える要求・制約・関心事を以下に分類する。

### 2.2.1 主要機能要求（Primary Functionality）

計画書セクション 3.1（P0: 最小必須機能）から抽出した、アーキテクチャに直接影響する主要機能を以下に示す。

| ID | 機能要求 | 計画書参照 | 関連ビジネス目標 |
|---|---|---|---|
| FR-01 | **Scenario Pack + Registry**: 実験1件をネットワーク定義・時系列データ・シミュレータ設定・評価指標・seed・expected outputs・可視化テンプレートを含むパッケージとして管理できる。Scenario Registry により登録・検索・バージョン管理ができる | 3.1 A, 5 | BG-1, BG-2, BG-3, BG-4 |
| FR-02 | **Orchestrator**: Docker ベースで各シミュレータ・解析系の統合実行（実行順序管理、コンテナ起動・管理、時間同期、バッチ実行、結果収集、エラー収集）ができる | 3.1 B | BG-1 |
| FR-03 | **Canonical Data Layer**: ツールごとの独自フォーマットを共通表現（topology, asset, timeseries, event, metric, experiment metadata）に変換し、CSV/JSON/Parquet でエクスポートできる | 3.1 C | BG-1, BG-4 |
| FR-04 | **Benchmark Harness**: 電圧逸脱率、thermal overload 時間、ENS、dispatch cost、CO2、curtailment、restoration time、runtime 等の評価指標で定量的に採点・比較できる | 3.1 D | BG-4 |
| FR-05 | **CLI + Notebook Bridge**: CLI で全操作が完結し、notebook/script で深掘りできる。Web UI は補助的な可視化手段とする | 3.1 E | BG-1, BG-2 |
| FR-06 | **段階的カスタムレイヤー**: L1（設定変更）〜L4（ソース改変）まで、研究者のスキルレベルに応じた拡張ができる。上位レベルが下位レベルの仕組みを壊さない | 3.1 F | BG-2, BG-4 |
| FR-07 | **Simulator Connectors**: OpenDSS、GridLAB-D、HELICS、pandapower、Grid2Op に対するアダプタを提供する | 5, 6 | BG-1 |

### 2.2.2 品質属性要求（Quality Attribute Scenarios）

計画書セクション 10.1（KPI）およびセクション 5（技術アーキテクチャ）から抽出し、ACDM の品質属性シナリオ形式（刺激源・刺激・成果物・環境・応答・応答測定値）で記述する。

#### QA-1: 導入容易性（Installability）

| 項目 | 内容 |
|---|---|
| 刺激源 | 新規ユーザー（研究者・研究室メンバー） |
| 刺激 | gridflow を初めてセットアップする |
| 成果物 | gridflow 実行環境 |
| 環境 | Windows / macOS / Linux、Docker Desktop インストール済み |
| 応答 | `docker compose up` で全スタックが起動する |
| 応答測定値 | **セットアップ完了時間 < 30 分**、**セットアップ成功率 > 90%** |

| 項目 | 内容 |

| 応答 | サンプル Scenario Pack を使って結果が得られる |

### 2.2.3 制約（Constraints）

アーキテクチャ設計において変更不可能な前提条件を以下に示す。

| ID | 制約 | 根拠 | 計画書参照 |
|---|---|---|---|
| CON-1 | **Python を唯一のアプリケーション言語とする** | MATLAB/Simulink/Octave に依存しないことで、ライセンスコスト・環境構築の障壁を排除する。pandapower が MATPOWER の機能を Python ネイティブでカバーしている | 5「言語方針」 |
| CON-2 | **Docker ベースのデプロイ環境** | ユーザーへの配布は Docker Compose を標準とし、`docker compose up` で全スタックが起動する設計とする。ホスト OS に直接依存する機能を持たない。ただし開発時は `pip install -e .` によるローカル Python 環境での開発・テストを許容する | 5.1「動作環境」 |
| CON-3 | **1 人 + AI 開発体制** | 1人開発 + Claude を前提とする。初期 6〜10 万 LOC。この制約がアーキテクチャの複雑さの上限を決定する | 1.3「開発体制」 |
| CON-4 | **AMD64 + ARM64 マルチアーキテクチャ対応** | Apple Silicon 対応必須。各ツールのアーキテクチャ対応状況に応じた環境戦略が必要 | 5.1「コンテナ内ランタイム」 |
| CON-5 | **OSS として公開** | Scenario Pack format、benchmark format、canonical schema、connector SDK を公開する。研究コミュニティへの普及を優先するため初期は全機能無料 | 9.2, 10.3 |
| CON-6 | **英語を標準言語とする** | ドキュメント・CLI メッセージ・エラーメッセージは英語を標準とする。海外展開を初期から視野に入れるため | 10.2 |

### 2.2.4 アーキテクチャ関心事（Architectural Concerns）

明示的な要求や制約ではないが、アーキテクチャ判断に影響を与える関心事を以下に示す。

#### AC-1: 既存 OSS との関係性（Wrapper → Hybrid → Full）

計画書セクション 8 で 3 つの選択肢（Wrapper-first / Hybrid absorption / Full superseding platform）が定義されている。アーキテクチャは**初期は Wrapper-first で入りつつ、Hybrid absorption に移行可能な構造**でなければならない。

具体的には:
- Connector 層が外部シミュレータを隠蔽し、交換可能であること
- Canonical Data Layer がツール非依存であること
- 将来的に solver 本体を自前実装に置き換えても、上位層（Orchestrator、Benchmark、CLI）に影響しないこと

判断基準（計画書 8.4）:
1. 既存 OSS の導入性が研究室への展開の障害になっている
2. 研究室ごとに同じ前処理・変換・採点コードを量産している
3. 既存 OSS を跨ぐことで結果の再現性が崩れる
4. 共通ベンチマークのための抽象化が既存 OSS では難しい
5. 外部依存の更新に振り回される

#### AC-2: 段階的な成長パス

コード規模の目安（計画書 7.1）:

| フェーズ | 想定規模 | 意味 |
|---|---|---|
| Phase 0 | 2〜4 万 LOC | 技術検証版 |
| Phase 1 | 6〜10 万 LOC | 研究導入可能な MVP |
| Phase 2 | 10〜15 万 LOC | 複数研究室導入 |
| Phase 3 | 12〜18 万 LOC | ワークフローエンジンとして定着 |
| Phase 4 | 20〜35 万 LOC | 上位互換統合プラットフォーム |

アーキテクチャは Phase 1 で成立し、Phase 4 まで破綻しない構造であることが求められる。

#### AC-3: 教材化可能な拡張性の確保

gridflow 自体は教育教材を直接提供しない（開発者に教育設計の専門知識がないため）。ただし、教員パートナーが Scenario Pack を授業用に加工・共有できる拡張性を確保する。

具体的には:
- Scenario Pack にメタデータ（難易度、前提知識、学習目標等）を付与できるスキーマ拡張性
- Scenario Pack の共有・再配布が容易な構造（GitHub ベース）
- grading / course pack は gridflow のコア責務外とし、外部ツールや教員の裁量に委ねる

#### AC-4: 複数研究トラック（Track 1-3）の同時サポート

計画書セクション 4 で 3 つの初期研究トラック（マイクログリッド運用 / DER hosting / 逐次運用）が定義されている。Canonical Data Layer と Benchmark Harness は、これら異なるドメインの評価指標を統一的に扱える設計が必要である。

#### AC-5: cache / resume の設計への影響

計画書セクション 6「責務分割」で Orchestrator の責務に cache と retry が含まれている。cache/resume は P1 機能（計画書 3.2）だが、Orchestrator の内部設計に大きく影響する。P0 段階で cache を後付け不可能な設計にしないよう、Orchestrator の実行状態管理に拡張余地を確保する必要がある。

#### AC-6: 継続利用への配慮

計画書 10.1 の KPI に「7 日リテンション率 > 50%」がある。リテンション率はアーキテクチャ単体で解決する問題ではなく、UX・コンテンツ・コミュニティの複合要因で決まる。アーキテクチャとしては、利用状況の計測基盤（opt-in テレメトリ）の設計余地を確保するに留める。
