# 2. アーキテクチャの重要事項

## 2.1 ビジネス目標

計画書（セクション 0, 1.2, 2.1, 9.4, 9.5）から抽出したビジネス目標を以下に示す。

### BG-1: 電力系研究の E2E ループを高速化する

> **研究テーマではなく、研究の回し方を高速化する**（計画書 2.1）

電力系研究室が繰り返し払っている実験運用コスト（環境構築、連携コード作成、入出力整形、実験再実行、グラフ生成、表整理）を圧縮するワークフローエンジンを提供する。

### BG-2: 共同研究の再現実験基盤として普及する

> **既存論文の再現パックを一緒に作りませんか**（計画書 9.4）

初期導入先は大学研究室（国内外）および共同研究の再現実験基盤である（計画書 2.3）。導入研究室数を主要指標とし、マイルストーンとして 10 → 50 → 100 研究室を目指す（計画書 10.2）。

教育利用（授業演習・grading 等）は gridflow の直接の責務としない。研究者が自分の研究室の学生に使わせる形で自然に広がることを想定し、教員パートナーが教材化できる拡張性を確保するに留める。

### BG-3: 実験の再現性を制度として担保する

卒業と同時に実験が失われる問題を防ぎ、実験条件の再現性を Scenario Pack として制度化する（計画書 3.1 A）。

### BG-4: 研究フローと評価方法を標準化する

研究手法ではなく、研究フローと評価方法を標準化する（計画書 9.5）。共通ベンチマークによる定量比較を可能にする。

### 2.2.4 アーキテクチャ関心事（Architectural Concerns）

明示的な要求や制約ではないが、アーキテクチャ判断に影響を与える関心事を以下に示す。

#### AC-1: 既存 OSS との関係性（Wrapper → Hybrid → Full）

計画書セクション 8 で 3 つの選択肢（Wrapper-first / Hybrid absorption / Full superseding platform）が定義されている。アーキテクチャは**初期は Wrapper-first で入りつつ、Hybrid absorption に移行可能な構造**でなければならない。

具体的には:
- Connector 層が外部シミュレータを隠蔽し、交換可能であること
- Canonical Data Layer がツール非依存であること
- 将来的に solver 本体を自前実装に置き換えても、上位層（Orchestrator、Benchmark、CLI）に影響しないこと

判断基準（計画書 8.4）:
1. 既存 OSS の導入性が研究室への展開の障害になっている
2. 研究室ごとに同じ前処理・変換・採点コードを量産している
3. 既存 OSS を跨ぐことで結果の再現性が崩れる
4. 共通ベンチマークのための抽象化が既存 OSS では難しい
5. 外部依存の更新に振り回される

#### AC-2: 段階的な成長パス

コード規模の目安（計画書 7.1）:

| フェーズ | 想定規模 | 意味 |
|---|---|---|
| Phase 0 | 2〜4 万 LOC | 技術検証版 |
| Phase 1 | 6〜10 万 LOC | 研究導入可能な MVP |
| Phase 2 | 10〜15 万 LOC | 複数研究室導入 |
| Phase 3 | 12〜18 万 LOC | ワークフローエンジンとして定着 |
| Phase 4 | 20〜35 万 LOC | 上位互換統合プラットフォーム |

アーキテクチャは Phase 1 で成立し、Phase 4 まで破綻しない構造であることが求められる。

#### AC-3: 教材化可能な拡張性の確保

gridflow 自体は教育教材を直接提供しない（開発者に教育設計の専門知識がないため）。ただし、教員パートナーが Scenario Pack を授業用に加工・共有できる拡張性を確保する。

具体的には:
- Scenario Pack にメタデータ（難易度、前提知識、学習目標等）を付与できるスキーマ拡張性
- Scenario Pack の共有・再配布が容易な構造（GitHub ベース）
- grading / course pack は gridflow のコア責務外とし、外部ツールや教員の裁量に委ねる

#### AC-4: 複数研究トラック（Track 1-3）の同時サポート

計画書セクション 4 で 3 つの初期研究トラック（マイクログリッド運用 / DER hosting / 逐次運用）が定義されている。Canonical Data Layer と Benchmark Harness は、これら異なるドメインの評価指標を統一的に扱える設計が必要である。

#### AC-5: P1/P2 機能の設計への先行的影響

P0 のアーキテクチャ設計時に、以下の P1/P2 機能が後付け不可能にならないよう拡張ポイントを確保する必要がある。

| P1/P2 機能 | 影響を受ける P0 コンポーネント | 必要な設計配慮 |
|---|---|---|
| cache / resume（P1） | Orchestrator | 実行状態の永続化インターフェース。ステップ単位の中間結果保存ポイント |
| record / replay（P1） | Orchestrator, CDL | 実行イベントのログ可能な構造。CDL への書き込みがイベントとして記録可能 |
| experiment diff（P1） | Scenario Pack, CDL | Scenario Pack・結果のバージョニングと差分比較可能なデータ構造 |
| sensitivity sweep（P1） | Orchestrator, Scenario Pack | パラメータ空間の定義と batch 実行のパラメタライズ |
| fault injection（P1） | Orchestrator, Connector | 実行パイプラインへの介入ポイント（Connector の入出力をフックできる構造） |
| profiling（P1） | Orchestrator | 各ステップの実行時間・リソース使用量の計測ポイント |
| HIL 連携（P2） | Connector | リアルタイム制約のある Connector インターフェース。非同期通信の拡張余地 |
| cyber co-simulation（P2） | Orchestrator, Connector | 複数ドメイン（電力 + 通信）の時間同期拡張 |
| IEC 61850 / DNP3（P2） | Connector, CDL | 標準プロトコルのデータモデルと CDL のマッピング拡張 |

#### AC-6: 継続利用への配慮

計画書 10.1 の KPI に「7 日リテンション率 > 50%」がある。リテンション率はアーキテクチャ単体で解決する問題ではなく、UX・コンテンツ・コミュニティの複合要因で決まる。アーキテクチャとしては、QA-8（可観測性）の計測基盤の一環として、利用状況の opt-in テレメトリの設計余地を確保するに留める。

#### AC-7: 電力系規格との整合性

AS-4（シミュレータと実系統の非区別）により、gridflow のアーキテクチャは将来的に実系統制御にも適用可能である。この設計方針を活かすため、電力系の主要規格との整合性をアーキテクチャレベルで確保する。

##### 関連規格と影響を受けるコンポーネント

| 規格 | 内容 | 影響を受けるコンポーネント | 設計配慮 |
|---|---|---|---|
| **CIM（IEC 61970/61968）** | 電力系統の共通情報モデル。系統構成・設備・計測のデータ交換標準 | CDL | CDL のドメインモデル（topology, asset 等）が CIM のクラス構造と互換性を持つよう設計する。完全準拠ではなく、CIM との双方向マッピングが可能な構造にする |
| **IEC 61850** | 変電所自動化の通信プロトコル。GOOSE、MMS、Sampled Values | Connector | IEC 61850 対応は Connector 実装の問題。Connector インターフェースが IEC 61850 のデータモデル（Logical Node 等）を表現できる汎用性を持たせる |
| **IEC 60870-5-104** | 遠方監視制御プロトコル | Connector | 104 プロトコル対応の Connector 実装が載せられるインターフェース設計 |
| **DNP3** | 配電自動化プロトコル（北米中心） | Connector | IEC 104 と同様、Connector 実装レベルで対応 |
| **IEEE テストフィーダー** | 標準的なベンチマーク系統（IEEE 13/34/123 バス等） | Scenario Pack | 標準テストフィーダーを Scenario Pack として提供。ベンチマーク比較の共通基盤とする |

##### 設計方針

- **P0 段階:** CDL のドメインモデルを CIM と「対応関係を説明できる」粒度で設計する。完全な CIM 準拠は求めないが、CIM のクラス構造との対応表を作成できる程度の互換性を目標とする
- **将来:** CIM import/export の Connector 実装、IEC 61850 / 104 / DNP3 対応の Connector 実装を追加する。これらは AS-4 により、シミュレータ Connector と同じインターフェースの別実装として自然に追加できる
- **収益化への布石:** 規格準拠の検証ドキュメント・適合性テストスイート・規格対応ガイドは将来的に有償提供の候補となる。OSS として公開するコア機能（CON-5）とは別に、規格系ドキュメンテーションを付加価値として位置づける

## 2.3 アーキテクチャ戦略・方針（Architecture Strategies）

アーキテクチャドライバーに応じた高レベルの設計方針を以下に示す。これらは Round 2 以降のアーキテクチャパターン・戦術（セクション 2.4）で具体化される。

### AS-1: DDD（Domain-Driven Design）によるドメインモデリング

**動機:** 電力系統研究は topology、asset、timeseries、event、metric 等の専門的なドメイン概念を持つ。研究者（ドメインエキスパート）とコード上の語彙を一致させ（Ubiquitous Language）、複雑な問題をドメイン境界に沿って分割対処する。

**方針:**
- Canonical Data Layer のデータモデルを、電力系研究者が自然に理解できるドメインモデルとして設計する
- Bounded Context により、サブシステム間（Orchestrator / Connector / Evaluation / UX）の責務境界を明確にする
- 各 Bounded Context 間のやりとりは明示的なインターフェース（Anti-Corruption Layer）を介する
- ドメインモデルは外部ツール・フレームワークに依存しない（依存方向の制御は AS-2 で規定）

**駆動するドライバー:** FR-03（CDL）, FR-04（Benchmark）, AC-4（複数研究トラック）, QA-4（拡張性）

### AS-2: Clean Architecture による依存方向の制御

**動機:** テスト容易な設計は良い設計であり、拡張性が高い設計である。Dependency Inversion により、ドメインロジックを外部依存（シミュレータ、DB、フレームワーク）から隔離し、Connector の差替え・テスト時のモック注入・Wrapper → Hybrid → Full の段階移行・実システム統合を、全てインターフェース実装の入替えで実現する。

**方針:**
- 依存方向は「外側 → 内側」に統一する（Frameworks/Drivers → Interface Adapters → Use Cases → Entities）
  - Entities: ドメインモデル（CDL のデータモデル、評価指標等）— 外部依存なし
  - Use Cases: ワークフローロジック（Orchestrator の実行制御、Benchmark の評価ロジック等）
  - Interface Adapters: Connector、CLI、Notebook Bridge、データエクスポート
  - Frameworks/Drivers: Docker、外部シミュレータ、ファイルシステム、DB
- 全てのサブシステム間境界はインターフェース（Protocol / ABC）で定義する
- 実装の差替え（テストモック、Connector 切替え、将来の自前 solver 置換）はインターフェースの実装交換で行う

**駆動するドライバー:** AC-1（OSS 段階移行）, QA-4（拡張性）, FR-07（Connector）, CON-3（1人+AI 開発）

### AS-3: TDD（Test-Driven Development）による設計品質の駆動

**動機:** テストを先に書くことで、インターフェースの設計が強制的に明確になり、AS-2 の依存方向の制御が自然に達成される。1人+AI 開発体制ではリグレッション検知が特に重要であり、テストスイートが仕様書・安全網・設計検証の三役を兼ねる。

**方針:**
- ドメインロジック（Use Cases, Entities）はテストファーストで開発する
- テストが書きにくい設計は、依存方向が乱れているシグナルとして扱い、設計を見直す
- Connector のテストは、インターフェースに対するモックテスト（単体）と、実シミュレータに対する統合テストの 2 層で構成する
- CI でテストスイートを自動実行し、全テスト通過を merge の前提条件とする

**駆動するドライバー:** CON-3（1人+AI 開発）, QA-3（再現性）, QA-4（拡張性）

### AS-4: シミュレータと実系統の非区別（Simulation-Real Equivalence）

**動機:** Connector インターフェースから見れば、シミュレータも実系統（SCADA、HIL テストベンチ等）も「コマンドを受けて状態を返すもの」である。この区別をアーキテクチャレベルで行わないことで、シミュレーション → 実機制御のシームレスな移行、HIL の自然な実現、Digital Twin 構成が追加の設計変更なしに可能になる。

**方針:**
- Connector インターフェースはシミュレータ/実機を区別しない。どちらも同じインターフェースの異なる実装である
- シミュレータと実系統の差異（リアルタイム制約、安全性、可逆性、データ品質）は Connector 実装の内部で対処する。Orchestrator や CDL のレベルでは意識しない
- Scenario Pack も対象がシミュレータか実機かを区別しない。同一の Scenario Pack をシミュレータ Connector と実機 Connector の両方で実行可能
- P0 段階ではシミュレータ Connector のみ実装するが、インターフェース設計は実機 Connector も載せられる汎用性を持たせる

**表向きの位置づけ:**
gridflow は「電力系統の研究ワークフローエンジン」として市場に出す。アーキテクチャがシミュレーションと実系統制御を区別しないことは、ユーザーが自然に気づくことを期待し、積極的には主張しない。

**駆動するドライバー:** AC-1（OSS 段階移行）, FR-07（Connector）, QA-4（拡張性）, AC-5（P1/P2 先行影響: HIL 連携）

### AS-1〜AS-4 の関係

```
AS-1 (DDD)              AS-2 (Clean Architecture)     AS-3 (TDD)
ドメインを正しくモデル化 → 依存方向を制御して隔離   → テストで設計品質を駆動

  Ubiquitous Language       Dependency Inversion          Test-First
  Bounded Context           Interface Segregation         Red-Green-Refactor
  Anti-Corruption Layer     Ports & Adapters              テスト容易性 = 良い設計

                    AS-4 (Simulation-Real Equivalence)
                    Connector の背後を区別しない設計
                    AS-2 の DI が AS-4 を自然に実現する
```

- DDD が「何をモデリングするか」を決める
- Clean Architecture が「依存をどう制御するか」を決める
- TDD が「設計が正しく分離されているか」を検証する
- Simulation-Real Equivalence が「Connector の背後を限定しない」ことを保証する
