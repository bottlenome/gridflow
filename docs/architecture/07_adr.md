# 7. 設計判断の記録（Architecture Decision Records）

各 ADR は「決定済み」「保留」のステータスを持つ。保留の ADR は未解決事項（6.4）と対応する。

---

## ADR-001: 実装言語選択

| 項目 | 内容 |
|---|---|
| **ステータス** | **決定済み** |
| **コンテキスト** | CON-1 で MATLAB/Simulink/Octave 非依存を制約として定めた。実装言語の候補は Python / Rust / ハイブリッド |
| **候補** | (A) Python (B) Rust (C) Python + Rust ハイブリッド |
| **判断材料** | 3.4.3（言語選択への示唆）: コアは CPU バウンドではなく、ボトルネックは外部ツール側。性能より**エコシステム適合性**と**L2 Plugin 記述容易性**が決定要因 |
| **決定** | **(A) Python** |
| **根拠** | (1) 外部ツールのバインディングが全て Python 対応（dss-python, helics-python, pandapower, Grid2Op）。(2) 研究者の標準言語であり L2 Plugin（QA-4: < 100行）が現実的。(3) CON-3（1人+AI 開発）で最速の開発速度。(4) QA-10 分析でコアは CPU バウンドではなく、Python の性能で十分。(5) 将来ホットパスのみ Rust に段階移行可能（AS-2: インターフェース越しの差替え） |
| **Python バージョン** | 3.11+ |
| **型ヒント** | 全コードで型ヒント（type hints）を使用。mypy / pyright で静的型チェックを CI に組み込む |
| **段階移行パス** | QA-10 のオーバーヘッドが 5% を超えた場合、Connector 実装や CDL の I/O 部分を Rust（PyO3 経由）に段階移行 |
| **関連** | CON-1, 3.4.3, QA-4, QA-10 |

## ADR-002: Docker ベースの実行環境

| 項目 | 内容 |
|---|---|
| **ステータス** | **決定済み** |
| **コンテキスト** | QA-1（導入容易性 < 30 分）、QA-3（再現性）、QA-7（ポータビリティ）を同時に達成する実行環境が必要 |
| **候補** | (A) ネイティブインストール (B) Docker Compose (C) Kubernetes |
| **決定** | **(B) Docker Compose** |
| **根拠** | ネイティブは OS 差異で再現性が崩れる（QA-3）。Kubernetes は CON-3（1人+AI 開発）に合わない。Docker Compose が導入容易性・再現性・移植性の最良バランス |
| **関連** | CON-2, CON-4, QA-1, QA-3, QA-7 |

## ADR-003: CLI ファースト、Web UI は補助

| 項目 | 内容 |
|---|---|
| **ステータス** | **決定済み** |
| **コンテキスト** | ユーザーインターフェースの主軸を CLI にするか Web UI にするか |
| **候補** | (A) Web UI 中心 (B) CLI 中心 + Web 補助 (C) CLI のみ |
| **決定** | **(B) CLI 中心 + Web 補助** |
| **根拠** | CLI は自動化（CI/CD）・再現性（コマンド履歴）・LLM 連携（QA-9: UC-10）に適する。Web UI は可視化の補助手段として将来追加可能。CON-3 で両方を同時に作る余力はない |
| **関連** | FR-05, QA-9, CON-3 |

## ADR-004: Scenario Pack を実験の基本単位とする

| 項目 | 内容 |
|---|---|
| **ステータス** | **決定済み** |
| **コンテキスト** | 実験の再現性（QA-3）と共有性（BG-2）を制度として保証する仕組みが必要 |
| **候補** | (A) コードベースで実験管理（スクリプト + README） (B) Scenario Pack（宣言的パッケージ） (C) DB ベースの実験管理 |
| **決定** | **(B) Scenario Pack** |
| **根拠** | コードベースは暗黙知が多く再現性が崩れる。DB は導入障壁が高い（QA-1）。Scenario Pack は YAML（M-8）でバージョン管理可能、Git で共有可能（BG-2）、L1 研究者でも編集可能（QA-4） |
| **関連** | FR-01, BG-2, BG-3, QA-3, QA-4 |

## ADR-005: Canonical Data Layer によるツール間データ統一

| 項目 | 内容 |
|---|---|
| **ステータス** | **決定済み** |
| **コンテキスト** | 複数の外部ツール（OpenDSS, HELICS 等）の出力形式が異なる。比較・評価のために統一データ表現が必要 |
| **候補** | (A) ツール固有のデータをそのまま保存し、比較時に変換 (B) CDL（共通データ層）に変換して保存 |
| **決定** | **(B) CDL** |
| **根拠** | (A) は比較のたびに変換が必要で、N ツール × N ツールの変換マトリクスが発生する。CDL は N 回の変換（各 Connector 内の Anti-Corruption Layer）で統一でき、O(N) に削減。CIM（AC-7）との互換性も CDL 層で一元管理 |
| **関連** | FR-03, AC-7, AS-1 |

## ADR-006: Connector アーキテクチャ — 統一インターフェース

| 項目 | 内容 |
|---|---|
| **ステータス** | **決定済み** |
| **コンテキスト** | 外部ツールの接続方式が多様（インプロセス、サブプロセス、コンテナ間、リアルタイム）。3.1.3 で 5 ツールの特性を分析 |
| **候補** | (A) ツールごとに個別 I/F (B) 統一 I/F (C) 抽象レイヤー + 特殊化 |
| **決定** | **(B) 統一 I/F** |
| **根拠** | 3.1.3 で詳述。時間管理の違いは execute() 内部で吸収。AS-4（Simulation-Real Equivalence）、AS-2（DI）、AC-1（段階移行）、CON-3（シンプルさ）の全てと整合 |
| **関連** | FR-07, AS-4, AS-2, AC-1, 3.1.3 |

## ADR-007: 段階的カスタムレイヤー（L1-L4）

| 項目 | 内容 |
|---|---|
| **ステータス** | **決定済み** |
| **コンテキスト** | 研究者のスキルレベルに応じた拡張性が必要。拡張の敷居 = 導入の敷居（3.5.1） |
| **候補** | (A) 全機能をプログラミングで提供 (B) L1-L4 の段階的レイヤー |
| **決定** | **(B) L1-L4** |
| **根拠** | L1（YAML 編集のみ）で始められ、L2（Plugin API: < 100 行）で研究の大半をカバー。上位レベルが下位の仕組みを壊さない設計原則（3.5.4）。具体的な実現方法は 3.5.3 で例示 |
| **関連** | FR-06, QA-4, 3.5 |
