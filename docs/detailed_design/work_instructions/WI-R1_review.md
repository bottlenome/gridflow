## レビュー観点（チェックリスト形式で報告すること）

### A. 構造品質
- [ ] A1: 更新履歴が冒頭にあるか
- [ ] A2: 関連要件ID（REQ-xxx）が各セクション冒頭に明記されているか
- [ ] A3: Mermaid図が含まれているか（classDiagram/stateDiagram/erDiagram等）
- [ ] A4: 目次構成が計画書と一致しているか（セクション番号の欠落・順序違いがないか）

### B. 設計原則
- [ ] B1: Clean Architecture の依存方向が正しいか（外側→内側のみ。Domain層が外部に依存していないか）
- [ ] B2: Protocol（インターフェース）が消費側レイヤーに定義されているか（DIP原則）
- [ ] B3: IPO形式（Input→Process→Output）でメソッドが定義されているか（第3章のみ）
- [ ] B4: 単一責任原則: 1クラスが複数の責務を持っていないか

### C. 整合性
- [ ] C1: クラス名・モジュール名が第1章(01_requirements.md)1.4節の対応表と一致しているか
- [ ] C2: エラーコードがE-{レイヤー2桁}{連番3桁}体系（E-10xxx〜E-40xxx）に統一されているか
- [ ] C3: 要件ID（REQ-xxx）の参照が正しいか（存在しないIDを参照していないか）
- [ ] C4: 状態名・イベント名が章間で一致しているか（例: 第3章のOrchestrator状態と第5章の状態遷移図）

### D. 完成度
- [ ] D1: セクションが途中で切れていないか（「後続で追記」等の未完了マークがないか）
- [ ] D2: 表に空行・欠落がないか
- [ ] D3: Mermaid図の構文が正しいか（閉じ忘れ等）

## 報告フォーマット

ファイルは変更しないこと。以下の形式で報告:

```
### {ファイル名}
- ✅ A1, A2, A3, B1, B3, C1, C2, D1, D2, D3（問題なし項目をまとめて列挙）
- ⚠️ B2: ConnectorInterfaceがAdapter層に定義されている。UseCase層に移すべき（WARN）
- ❌ C3: REQ-F-099は存在しないIDを参照（ERROR）
```

問題なし項目は「✅」でまとめて列挙し、問題があるものだけ個別に報告すること。簡潔に。
