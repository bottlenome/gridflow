# 第7章 性能設計

本章では、gridflow の性能要件を整理し、ボトルネックの分析と対策方針、スケーラビリティの設計方針を示す。

## 7.1 性能要件一覧

| 要件 ID | 項目 | 目標値 | 根拠 |
|---|---|---|---|
| REQ-Q-010 | gridflow オーバーヘッド | 外部ツール実行時間の 5% 未満 | QA-10: ワークフローエンジン自体がボトルネックにならないこと |
| REQ-Q-010 | メモリ使用量 | ネットワーク規模に対して線形または準線形増加 | 大規模系統での利用を阻害しないこと |
| REQ-F-002 | バッチ実行 | 感度解析スイープにおける複数バリアント逐次実行 | Orchestrator によるバッチ実行機能 |

### 計測方針

- `StructuredLogger.timed()` コンテキストマネージャにより、各ステップの所要時間を JSON ログへ記録する
- 外部シミュレータ実行時間と gridflow 処理時間を分離計測し、オーバーヘッド比率を算出する
- メモリ使用量はプロセスの RSS を定期サンプリングし、ネットワークノード数との相関を確認する

## 7.2 ボトルネック分析と対策方針

### 主要ボトルネック

| # | ボトルネック | 発生箇所 | 影響度 |
|---|---|---|---|
| 1 | Connector I/O | 外部シミュレータとのデータ送受信 | 高 — 全体実行時間の大部分を占める |
| 2 | CDL シリアライゼーション/デシリアライゼーション | Canonical Data Layer の変換処理 | 中 — ネットワーク規模に比例 |
| 3 | Docker コンテナ起動 | コンテナの初期化・プロセス立ち上げ | 中 — 特に短時間シミュレーションで顕著 |
| 4 | ログ I/O | 構造化ログのファイル書き込み | 低 — 非同期化で軽減可能 |

### 対策方針

| # | 対策 | 対象ボトルネック | 実装方針 |
|---|---|---|---|
| 1 | ストリーミングデータ転送 | Connector I/O | 結果を一括ではなくストリームで受信し、CDL 変換をパイプライン化する |
| 2 | 遅延読み込み（Lazy Loading） | CDL シリアライゼーション | 必要なデータのみをオンデマンドでロードし、不要なデシリアライゼーションを回避する |
| 3 | コンテナプリウォーミング | Docker 起動 | バッチ実行時に次バリアントのコンテナを事前起動し、待機時間を隠蔽する |
| 4 | 非同期ログ出力 | ログ I/O | `QueueHandler` によりログ書き込みをバックグラウンドスレッドへ委譲し、メインスレッドをブロックしない（`REQ-Q-008`） |

## 7.3 スケーラビリティ設計

### 現行設計（P0）

- **バッチ実行**: Orchestrator が感度解析スイープの複数バリアントを逐次実行する。各バリアントは独立した Scenario Pack パラメータセットで構成される（`REQ-F-002`）
- **CDL ストレージ**: ファイルベース（CSV/JSON/Parquet）で保存し、ディスク容量に応じてスケールする。データベースのオーバーヘッドを回避する（`REQ-C-001`）
- **メモリ管理**: CDL データはバリアント単位でロード・解放し、同時に保持するデータ量を制限する

### 将来拡張（P1 以降）

| 拡張項目 | 方針 | 対象フェーズ |
|---|---|---|
| 並列 Scheduler | バリアント間の並列実行を Scheduler レイヤーで管理。コンテナ間の依存関係を DAG として解決 | P1 |
| 分散実行 | 複数ホストへのコンテナ分散配置（Docker Swarm / Kubernetes） | P2 |
| CDL キャッシュ | 変換済み CDL の中間結果キャッシュによるリラン高速化 | P1 |

### スケーラビリティ制約

- 1 人 + AI 開発体制のため、複雑な分散システムは P0 スコープ外とする（`REQ-C-001`）
- P0 では単一ホスト・逐次実行を前提とし、性能要件 `REQ-Q-010` を満たすことを優先する
