## 9.1 脅威モデル

### スコープ前提

| 項目 | 前提 |
|---|---|
| 利用環境 | 研究室内ローカル環境（`REQ-C-002`） |
| ユーザー | 信頼されたユーザー（研究室メンバー） |
| ネットワーク | ローカル Docker ネットワーク |
| マルチテナント | 非対応 — 単一ユーザーまたは単一研究室での利用を想定 |

本システムは商用ユーティリティ向けのプロダクション環境ではなく、研究室での利用を前提とする。そのため、セキュリティ設計はリスクに見合った適切な水準とする。

### 識別された脅威

| # | 脅威 | リスク | 対策方針 |
|---|---|---|---|
| T-1 | 悪意のある Scenario Pack | 中 | Pack 内のファイルに対するスキーマ検証。任意コード実行パスの排除 |
| T-2 | コンテナエスケープ | 低 | non-root 実行、ファイルシステム制限、ネットワーク分離 |
| T-3 | 依存関係サプライチェーン攻撃 | 低 | Docker イメージのダイジェストピン留め、Python 依存のロックファイル管理 |
| T-4 | 共有ボリューム経由の意図しないデータアクセス | 低 | ファイルパーミッション制御、マウントポイントの最小化 |

## 9.2 コンテナセキュリティ

### non-root コンテナ実行

- すべての gridflow 管理コンテナは非 root ユーザーで実行する
- Dockerfile にて `USER` ディレクティブで専用ユーザーを指定する
- ホスト側の特権エスカレーションリスクを低減する

### ファイルシステム制限

- コンテナのルートファイルシステムは可能な限り `read-only` でマウントする
- 書き込みが必要なパス（CDL 出力、一時ファイル）のみ `tmpfs` またはバインドマウントで提供する
- 不要なバイナリ・ツールを含まない最小イメージを使用する

### ネットワーク分離

- コンテナ間の通信はユーザー定義 Docker ネットワーク内に限定する
- 外部ネットワークアクセスが不要なコンテナは `--network=none` で起動する
- シミュレータコンテナ間の不要な通信を防止する

### イメージピン留め

- Docker イメージはタグではなくダイジェスト（SHA256）で指定する（`REQ-C-005`）
- イメージのダイジェストを Scenario Pack メタデータに記録し、再現性にも寄与する（`REQ-Q-003`）

## 9.3 データ保護方針

### データ分類

| データ種別 | 機密度 | 保護方針 |
|---|---|---|
| 実験データ（CDL 出力） | 低 — 研究データであり PII を含まない | ファイルパーミッションによるアクセス制御 |
| Scenario Pack | 低 | スキーマ検証、ハッシュによる改竄検出 |
| Connector クレデンシャル | 中 | Docker secrets による管理 |
| ログファイル | 低 | ファイルパーミッション制御 |

### 共有ボリュームの保護

- 共有ボリュームのファイルパーミッションを適切に設定する（owner: gridflow ユーザー、group: 研究室グループ）
- コンテナからのマウントは必要最小限のパスに限定する
- 書き込み権限は出力ディレクトリのみに付与する

### Connector クレデンシャル管理

- 外部サービスへの認証情報は環境変数や設定ファイルに直接記載しない
- Docker secrets を使用し、コンテナ内の `/run/secrets/` 経由でアクセスする
- クレデンシャルはログ出力から自動的にマスクする

### 将来拡張

| 拡張項目 | 方針 | 対象フェーズ |
|---|---|---|
| 保存時暗号化 | 機密性の高いシナリオデータに対して暗号化を提供 | P2 |
| 監査ログ | セキュリティ関連イベントの専用ログ出力 | P1 |
| Scenario Pack 署名 | Pack の出所を検証するデジタル署名 | P2 |
