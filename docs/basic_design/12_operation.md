## 12.1 運用想定

**関連要求:** REQ-C-003 (Docker Desktop 単一マシン), REQ-C-005 (研究室環境)

### 対象環境

gridflow は研究室環境での利用を前提とし、24/7 サーバ運用は想定しない。

| 項目 | 想定値 |
|---|---|
| 利用マシン数 | 1 台 / 研究室 |
| 同時利用者数 | 1 - 10 名 / 研究室 |
| 利用パターン | 間欠的（実験実行時のみ起動） |
| 実行環境 | Docker Desktop（Windows / macOS / Linux） |
| 可用性要件 | 非ミッションクリティカル（研究業務時間内に動作すれば十分） |

### 運用モデル

```
研究者 PC（Docker Desktop）
  └─ docker compose up（実験時に起動）
       ├─ gridflow コア
       ├─ Connector コンテナ群
       └─ (オプション) 可視化サーバ
  └─ docker compose down（実験終了後に停止）
```

- 常時起動は不要。`gridflow run` コマンド実行時に必要なコンテナが自動起動
- リソース制限: Docker Desktop のリソース設定に依存（推奨: CPU 4+, Memory 8GB+）
- データ永続化: Docker volume + ホストマシンのプロジェクトディレクトリ

## 12.2 監視・アラート方針

**関連要求:** REQ-Q-008 (依存関係最小化), REQ-C-003 (Docker Desktop 環境)

### 基本方針

P0 フェーズでは外部監視スタックを導入せず、ファイルベースの構造化ログと CLI ツールで運用する。

### ログ設計

| ログ種別 | 形式 | 出力先 | 保持期間 |
|---|---|---|---|
| アプリケーションログ | 構造化 JSON (jsonlines) | `~/.gridflow/logs/` | ユーザ設定（デフォルト 30 日） |
| 実験実行ログ | 構造化 JSON | Scenario Pack 結果ディレクトリ内 | 実験結果と同期間 |
| Docker コンテナログ | Docker 標準出力 | Docker ログドライバ | Docker 設定に依存 |

### KPI モニタリング

`gridflow metrics` CLI コマンドで主要 KPI を確認可能。

```bash
gridflow metrics                    # 直近の実行統計サマリ
gridflow metrics --detail           # 詳細メトリクス
gridflow metrics --export csv       # CSV エクスポート
```

| メトリクス | 内容 | 対応要求 |
|---|---|---|
| 実行時間 | シミュレーション実行時間とオーバーヘッド | REQ-Q-010 |
| セットアップ時間 | 環境構築所要時間の記録 | REQ-Q-001 |
| 再現性スコア | マルチ実行間の結果差異 | REQ-Q-003 |
| エラー率 | 実行失敗率とエラー分類 | REQ-Q-009 |

### テレメトリ

- **デフォルト: 無効（オプトイン方式）**
- `gridflow config set telemetry.enabled true` で有効化
- 収集データ: 匿名化された利用統計（コマンド実行頻度、エラー種別、OS/アーキテクチャ）
- 個人情報・実験データは収集しない
- 送信先・収集データの詳細はドキュメントで公開

## 12.3 保守プロセス

**関連要求:** REQ-C-005 (OSS 開発), REQ-Q-008 (依存関係最小化)

### OSS メンテナンス体制

| プロセス | 運用方法 |
|---|---|
| Issue 管理 | GitHub Issues（バグ報告、機能要望、質問のテンプレート提供） |
| PR レビュー | 最低 1 名のメンテナによるレビュー必須 |
| ブランチ戦略 | GitHub Flow（main + feature ブランチ） |
| コード品質 | ruff + mypy/pyright による自動チェック（CI 必須パス） |

### 依存関係管理

| 項目 | 方針 |
|---|---|
| 依存更新ツール | Dependabot（GitHub 標準機能） |
| 更新頻度 | セキュリティパッチ: 即時、メジャー更新: 月次レビュー |
| ロックファイル | `uv.lock` でバージョン固定 |
| 脆弱性スキャン | GitHub Security Advisories + `uv pip audit` |

### リリースサイクル

| リリース種別 | 頻度 | 内容 |
|---|---|---|
| PATCH (x.x.X) | 必要に応じて随時 | バグ修正、セキュリティパッチ |
| MINOR (x.X.0) | 月次（目安） | 後方互換な機能追加 |
| MAJOR (X.0.0) | 必要時のみ | 破壊的変更（移行ツール必須提供） |

### 破壊的変更ポリシー

1. MAJOR バージョンでのみ破壊的変更を許容
2. 最低 1 MINOR バージョン前に非推奨（deprecation）警告を導入
3. Migrator コンポーネントによる自動移行ツールを同時提供（REQ-Q-007 参照）
4. 移行ガイドをリリースノートに含める

### インシデント対応

| 重大度 | 定義 | 対応目安 |
|---|---|---|
| Critical | データ損失・セキュリティ脆弱性 | 24 時間以内に PATCH リリース |
| High | 主要機能の動作不能 | 1 週間以内に PATCH リリース |
| Medium | 一部機能の不具合 | 次回 MINOR リリースに含める |
| Low | 軽微な不具合・改善要望 | バックログとして管理 |
