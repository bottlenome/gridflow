# 課題深掘りの連鎖 — 候補 #1 を例に

ユーザー指摘: 「L1-L8 の抽象 scale ではわかりにくい。**どういう課題を、どういう方向で詳細化すると、何が見えてきたか** で書いて」

そのとおりなので書き直す。各段階は **「前の段階の課題 → 何を詳細化したか → 出てきた新しい課題」** の 3 点セット。

---

## 連鎖の全体像 (1 行ずつ)

```
S0  HCA の出力は不確実性を持つ
   ↓ [どんな構造の不確実性かを観察した]
S1  ある条件下では HCA 出力の "ばらつき" が完全に消える (stdev=0) regime がある
   ↓ [なぜ消えるのかを式で書いた]
S2  PV 注入電圧上昇が base 電圧降下より小さいとき、PV 配置を変えても違反集合が変わらない
   ↓ [この regime で誰が困るかを具体化した]
S3  HCA レポートを受領する配電事業者は「この HC=5 MW という数字が brittle か robust か」を判定したいが、現状は全部 stochastic で回して比べるしかない
   ↓ [事業者が直面する具体的な決定を書き出した]
S4  事業者は feeder を受け取った瞬間「この feeder で stochastic HCA (N=500) を回すべきか、deterministic (N=1) で十分か」を選ぶ必要がある
   ↓ [選択を間違えたときの損失を書き出した]
S5  False Negative (stochastic が必要なのに deterministic で済ませた) は規制 / 訴訟リスクで FP の数千倍重い、つまり非対称
   ↓ [この決定に使える資源を制約として書き下した]
S6  判定そのものは power flow を 1 回も走らせてはならない (= 走らせたら最初から stochastic 回せばよく無意味)
   ↓ [PF ゼロで使える特徴量を列挙した]
S7  使えるのは topology metrics と analytic ΔV (R·P/V_base) だけ。既存の HCA-ML は全部 PF を前提するので適用できない
   ↓ [事業者が信じる evidence の形を書き下した]
S8  評価は leave-one-feeder-out CV、判定基準は recall ≥ 0.9 ∧ precision ≥ 0.6、解釈性のため線形モデル
```

ここまで来ると **手法は logistic regression on power-flow-free features 一択**。ほかの候補 (GNN, surrogate, BNN, ...) は全部 S6 か S8 で落ちる。

---

## 連鎖の詳細 (各段階を 5 行で)

### S0 → S1: 「不確実性がある」を観察に下ろす

| 項目 | 内容 |
|---|---|
| 前段階の課題 | HCA の出力は stochastic input で揺れる (= 抽象スローガン) |
| 詳細化の方向 | 「揺れ方の **具体的構造** を実データで観察する」(数値に落とす) |
| やったこと | try9 で 1024 random PV placement の violation_ratio を集計 |
| 観察 | CIGRE MV @ load=1.0 で 256 件すべて violation_ratio = 0.600、stdev = 0.000 |
| 出てきた新課題 | "揺れる" 前提が成立しない regime が **存在** する。これは何? |

### S1 → S2: 観察を物理で説明する

| 項目 | 内容 |
|---|---|
| 前段階の課題 | stdev=0 regime がある (= 現象だけ) |
| 詳細化の方向 | 「物理機構を不等式で書き下す」 |
| やったこと | base 電圧降下と PV 電圧上昇のスケール比較 |
| 説明 | base voltages が 0.94 から ΔV_base = R·P_load/V > 0.06 pu 下がる一方、PV (50-500 kW) の上昇は ΔV_PV ≤ 0.005 pu。9/15 buses は PV 配置によらず < 0.94 pu |
| 出てきた新課題 | "違反集合が PV 配置に **不変** になる" 条件は load level に対して連続的に切り替わる。**どこで切り替わる?** |

### S2 → S3: 「だれが困るか」を具体化する

| 項目 | 内容 |
|---|---|
| 前段階の課題 | 切り替わる境界点が存在する (= 数学的事実) |
| 詳細化の方向 | 「この境界点を知らないと困る当事者を名前付きで挙げる」 |
| やったこと | 配電業界の役割を列挙 |
| 当事者 | (a) 配電事業者の **HCA レビュアー** (受領論文の数字をどう信じるか)、(b) **HCA SaaS 事業者** (1000 feeder/日の処理コスト)、(c) **規格委員会** (HCA の信頼区間 reporting standard) |
| 困らない人 | deterministic だけ使う伝統計画者、L2RPN/RL 系研究者 (HCA を直接扱わない) |
| 出てきた新課題 | この境界点を **事前に** 知る方法がない (= HCA を回した後でしか分散の有無が分からない、自己言及的問題) |

### S3 → S4: 当事者の決定を書き出す

| 項目 | 内容 |
|---|---|
| 前段階の課題 | 境界点を事前に知りたい (= 抽象 need) |
| 詳細化の方向 | 「**いつ・何を** 決めるか」を時系列で書き下す |
| やったこと | HCA SaaS 事業者の 1 日のオペレーションを想定 |
| 決定 | feeder の topology + base load を受け取った瞬間、「stochastic HCA (N=500、~3 分) を回す」or「deterministic load-flow (N=1、~50 ms)」を選ぶ。1000 feeder / 日 |
| 決定タイミング | power flow を 1 回も実行する前 |
| 出てきた新課題 | **判定器** が必要。入力 = (topology, base load)、出力 = stochastic 必要 / deterministic 十分 の binary |

### S4 → S5: 決定ミスのコストを書く

| 項目 | 内容 |
|---|---|
| 前段階の課題 | binary classifier を作る (= 普通の ML 問題に見える) |
| 詳細化の方向 | 「FP / FN それぞれの実害を **単位付き** で書き出す」 |
| やったこと | 規制・訴訟・計算予算で各エラーを書く |
| FN (stochastic 必要を見逃す) | 規制当局へ誤った robust 性主張 → 紛争・訴訟、政策誤判断。**コスト推定: 数百万円 〜 数億円 / 件** |
| FP (不要なのに stochastic 回す) | 計算 N=500 倍 = 1 feeder で 3 分が 25 時間。**コスト推定: ~100 円 / 件** (cloud compute) |
| 出てきた新課題 | 損失非対称 (FN >> FP)。**普通の accuracy 最大化でなく recall@precision=0.9 重視** の特殊な分類器設計 |

### S5 → S6: 使える資源を制約として書き出す

| 項目 | 内容 |
|---|---|
| 前段階の課題 | 非対称損失下の binary classifier (= まだ method 選択肢が広い) |
| 詳細化の方向 | 「判定に使える時間・データ・計算を厳密に列挙」 |
| やったこと | S4 のタイミングで利用可能な資源を確定 |
| 制約 1 | **power flow を 1 回も走らせない** (走らせるなら最初から stochastic 回せばよく判定の意味が消える、自己言及的矛盾) |
| 制約 2 | データは feeder topology graph + 推定 base load profile のみ (実測時系列はない) |
| 制約 3 | 判定は ≤ 1 秒 / feeder (実用要件、SaaS 事業者) |
| 制約 4 | 解釈性: 規格委員会が読める (linear / shallow tree) |
| 出てきた新課題 | これらを満たす分類器 method は何? |

### S6 → S7: 既存手法が制約に当てはめると消える

| 項目 | 内容 |
|---|---|
| 前段階の課題 | 制約付き binary classifier (= まだ複数 method ありそう) |
| 詳細化の方向 | 「method 候補を 5 つ列挙して制約で篩う」 |
| やったこと | HCA-ML 文献的な候補を 5 つ書いて消去法 |
| 候補 a (GNN with PF features) | 制約 1 違反 (PF 必要) → ❌ |
| 候補 b (Surrogate of stochastic HCA) | 制約 4 違反 (black box) → ❌ |
| 候補 c (Bayesian NN on topology) | 制約 4 違反 (解釈性) → ❌ |
| 候補 d (Causal graph + do-calculus) | 制約 3 違反 (1 秒以内困難) → ❌ |
| 候補 e (Logistic regression on analytic ΔV + topology metrics) | 全制約 OK → ✅ |
| 出てきた新課題 | **唯一**残った候補 e の特徴量集合 (= analytic ΔV と topology metrics の組合せ) は何が最適? |

### S7 → S8: 検証基準を当事者目線で書く

| 項目 | 内容 |
|---|---|
| 前段階の課題 | 候補 e の最適特徴量を選ぶ |
| 詳細化の方向 | 「事業者が **行動を変える** に足る evidence の形」 |
| やったこと | レビュアー (HCA SaaS 顧客 = 事業者) の意思決定基準を書き下す |
| 評価方式 | leave-one-feeder-out CV (K=7 feeder で 1 個ずつ hold-out) |
| 数値基準 | recall (true CV ≥ 0.1 を逃さない率) ≥ 0.9 かつ precision ≥ 0.6 |
| 図表 schema | confusion matrix at CV=0.1 threshold + feature importance bar + misclassified case 一覧 |
| 解釈性 | top-3 feature の係数を物理単位で書く (例: `+0.42 / pu_voltage_margin`) |
| 出てきた新課題 | これを満たすか実測する (= Phase 1 実験) |

## なぜこの **連鎖が** novelty を担保するのか

1. **S0** だけだと既存研究と区別できない (HCA 不確実性なんて 100 本書かれてる)
2. **S5** まで行くと「FN 重視の binary classifier」と読める。これも応用機械学習の枠で似たものはある
3. **S6** で初めて「**PF を 1 回も走らせない**」という他で見ない制約が立ち上がる。これがなぜ必要かは S4 タイミング (= PF 走らせる前) と S5 コスト (= PF 走らせたら判定の意味消失) からしか出てこない
4. **S7** で既存手法 a-d がこの制約 1 つで全部消える。**S6 を 1 つ緩めると 4 通りに分岐**。逆に言うと、S0〜S5 を共有する論文があっても S6 で分岐するので別問題

= **S0〜S6 の連鎖を辿らないと、S7 の手法 (PF-free linear classifier) には到達できない**。これが「課題深掘りによる手法強制」の意味。

## 連鎖の自己テスト (3 つの質問)

1. **連鎖の途中で抜けがある?**
   - S5 → S6 で「他に守るべき制約はないか?」を再検査 → コミュニケーション制約 (規格委員会読解性 = 制約 4) を追加した、これが候補 c の Bayesian NN を蹴る決定打
   - 抜けなし

2. **どこかで分岐があってサボった?**
   - S3 で「困らない人」も書くことで、本問題が伝統計画者 / L2RPN にとっては関係ないと明示 → スコープが正確
   - サボりなし

3. **最終手法に到達するのに、連鎖のどれが必須?**
   - S1 (現象観察) なしには「分散ゼロ regime」を知らない → S4 の決定問題が立ち上がらない
   - S5 (コスト非対称) なしには通常 accuracy 最大化分類器に流れる
   - S6 (PF 0 回) なしには既存 HCA-ML 系から区別できない
   - **S1, S5, S6 が同時に効いて初めて手法が一意化**
