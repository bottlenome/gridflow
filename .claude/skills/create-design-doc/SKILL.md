# 設計書作成スキル

ユーザーが設計書の作成を依頼したとき、このスキルに従って作成する。

## 入力の確認

まず以下を確認する:

1. **アーキテクチャドキュメント**の存在（`docs/architecture/` 等）
2. **計画書**の存在（プロダクト定義、機能要件）
3. 作成する設計書の種類（`$ARGUMENTS` で指定。デフォルト: 基本設計書）

入力ドキュメントが見つからない場合はユーザーに確認する。

## 作成手順

### Phase 1: 計画策定

1. アーキテクチャドキュメントと計画書を全て読み込む
2. IPA ガイドラインに基づく構成案を作成する
3. **作成計画書**（`docs/<design-type>_plan.md`）を作成し、ユーザーに提示する

作成計画には以下を含める:
- 構成（IPA 準拠の章立て）
- 着手手順（Phase 分割）
- 要求 ID 体系
- ファイル構成
- 更新ドキュメント管理方針

**ユーザーの承認を得てから Phase 2 に進む。**

### Phase 2: ドキュメント作成

#### 必須要素（全設計書共通）

1. **要求一覧**（第1章に配置）
   - ビジネス要求: `REQ-B-xxx`
   - 機能要求: `REQ-F-xxx`
   - 品質要求: `REQ-Q-xxx`
   - 制約: `REQ-C-xxx`
   - アーキテクチャドキュメントの BG/FR/QA/CON と対応付ける
   - **トレーサビリティマトリクス**（要求 → 設計章の対応表）

2. **更新履歴**
   - README に全体の更新履歴
   - 各章に章ごとの更新履歴
   - 付録に更新ドキュメント一覧

3. **目次**
   - README.md に全章へのリンク付き目次
   - 読み方ガイド

#### IPA 準拠の基本設計書構成（12章 + 付録）

```
docs/basic_design/
├── README.md                    # 目次・更新履歴
├── 01_requirements.md           # 第1章 要求一覧（トレーサビリティ）
├── 02_system_architecture.md    # 第2章 システム方式設計
├── 03_function_design.md        # 第3章 機能設計
├── 04_cli_design.md             # 第4章 画面設計（CLI/UI に読み替え）
├── 05_data_design.md            # 第5章 データ設計
├── 06_external_interface.md     # 第6章 外部インターフェース設計
├── 07_performance.md            # 第7章 性能設計
├── 08_reliability.md            # 第8章 信頼性設計
├── 09_security.md               # 第9章 セキュリティ設計
├── 10_deployment.md             # 第10章 移行・導入設計
├── 11_test_policy.md            # 第11章 テスト方針
├── 12_operation.md              # 第12章 運用・保守設計
└── appendix.md                  # 付録（用語集・参考文献・更新ドキュメント一覧）
```

#### IPA 項目とプロダクト特性の読み替え

| IPA 推奨項目 | CLI ツールの場合 | Web アプリの場合 |
|---|---|---|
| 画面設計 | CLI コマンド体系・入出力仕様 | 画面遷移図・ワイヤーフレーム |
| バッチ設計 | バッチ実行・ジョブ管理機能 | バッチ処理・スケジューラ |
| 帳票設計 | エクスポートフォーマット | レポート・帳票出力 |
| DB 設計 | データモデル（ファイルベースも可） | DB スキーマ設計 |

#### 各章の書き方

各章は以下のフォーマットで書く:

```markdown
# 第N章 タイトル

本章では...（章の目的を1-2文で記述）

## 更新履歴

| 版数 | 日付 | 変更内容 |
|---|---|---|
| 0.1 | YYYY-MM-DD | 初版作成 |

---

## N.1 セクション名

**関連要求:** REQ-F-xxx, REQ-Q-xxx

（本文。Mermaid 図、コード例、テーブルを適宜含める）
```

### Phase 3: 並行作成の戦略

効率化のため、以下の順序で並行作成する:

1. **最初に作成:** 第1章（要求一覧）+ README（目次）— 他の全章の基盤
2. **構造面を並行:** 第2章（システム方式）+ 第5章（データ）+ 第6章（外部IF）
3. **振る舞い面を並行:** 第3章（機能）+ 第4章（画面/CLI）
4. **非機能を並行:** 第7章（性能）+ 第8章（信頼性）+ 第9章（セキュリティ）
5. **導入・運用を並行:** 第10章（移行）+ 第11章（テスト）+ 第12章（運用）
6. **仕上げ:** 付録 + トレーサビリティ検証

Agent ツールを活用して、2-5 を並行実行する。

### Phase 4: 整合性確認

- 第1章のトレーサビリティマトリクスで全要求が対応されているか確認
- 入力ドキュメント（アーキテクチャドキュメント等）に更新履歴がなければ追加
- アーキテクチャドキュメントから基本設計書への参照を追加

### Phase 5: コミット・プッシュ

作成した全ファイルをコミットしてプッシュする。

## 品質基準

- [ ] 全要求に REQ-xxx ID が付与されている
- [ ] トレーサビリティマトリクスで全要求が少なくとも1つの章で対応されている
- [ ] 全章に更新履歴がある
- [ ] README に目次と読み方ガイドがある
- [ ] 付録に更新ドキュメント一覧がある
- [ ] Mermaid 図が適切に含まれている
- [ ] Python コード例が Interface/Protocol レベルで含まれている
- [ ] アーキテクチャドキュメントの用語・ID と一貫している
