# CLAUDE.md — gridflow プロジェクトルール

## 0. プロジェクトの設計哲学（最重要）

### 0.1 妥協なき理想設計の原則

**本プロジェクトは「妥協せず、理想の設計を一度で出す」ことを最優先とする。**

設計判断において以下を徹底する：

- **一貫性に例外を作らない**: 「この属性だけ特別扱い」「この章だけ別ルール」を作らない。原則を立てたら全箇所に適用する
- **責務分離の言い訳で原則を緩めない**: 「これは補助情報だから dict でいい」「これは内部実装だから可変でいい」のような例外を作らない。frozen と決めたら全 frozen、不変と決めたら全不変
- **後段の都合で前段を歪めない**: 実装の都合で Domain モデルを汚染しない。実装が辛いなら実装側で吸収する
- **整合性は段階的でなく一括で取る**: 「次のフェーズで直す」「いつかリファクタする」を避ける。気づいた時点で全章・全ファイルに反映する
- **歴史的経緯は記録するが、設計の純粋性は維持する**: 変更の経緯は更新履歴と review_record に残す。ただし「過去がこうだったから今もこう」という理由で妥協しない

### 0.2 なぜこの方針か

gridflow は研究系ツールであり、**再現性・等価性・ハッシュ可能性・型安全性**が生命線。これらは「ほとんど守られる」では意味がなく、**例外なく守られる**ことに価値がある。一箇所でも mutable な dict が混ざれば、その瞬間に Asset を `set` に入れられなくなり、実験ログから状態を完全復元できなくなる。

「ほぼ理想」と「理想」の差は、コード上は1行でも、保証の質においては無限大である。

### 0.3 適用例

- **論点6.1（parameters の型）**: `dict` → `tuple[tuple[str, object], ...]` に統一。「parameters だけ」ではなく `metadata` / `properties` / `params` も含めて全付帯属性を tuple 化（v0.x で適用）
- **論点6.4（StepResult）**: 「最小3属性で十分」ではなく、`step_id` / `timestamp` / `error` / Enum status まで一度で確定
- **論点6.3（PackNotFoundError）**: 「Domain に置くだけ」ではなく `ScenarioRegistry` を Domain Protocol 化してエラー契約まで明示
- **論点6.6（Orchestrator の責務分割）**: 「アーキテクチャ doc を Infra に直す（妥協案 B）」「詳細設計を UseCase に戻す（妥協案 A）」ではなく、**役割分割（C 案）** を選択。`Orchestrator` (UseCase, ビジネスロジック) と `ContainerOrchestratorRunner` (Infra, Docker 操作) を `OrchestratorRunner` Protocol で接続。実装影響が大きくても層境界を完全に守る

### 0.4 妥協の誘惑が来たときの対処

「ここは例外でいいかも」と思ったら以下を自問する：

1. その例外を作ることで、原則のどの不変条件が破れるか？
2. 例外を作らないと本当に困るのか？それとも作る側の楽のためか？
3. 5 年後、新規参加者がこのコードを見たとき、例外の理由を即座に理解できるか？

3 つすべてに「問題なく原則を貫ける」と答えられない限り、**例外を作らない**。

## サブエージェントへの指示ルール

### 1. 作業指示書（WI）方式を使う

サブエージェントに直接「基本設計書を読んで」と指示しない。アーキテクト（メインエージェント）が必要情報を抽出し、作業指示書ファイルに埋め込む。

```
❌ 悪い例: 「/home/user/gridflow/docs/basic_design/03_function_design.md を読んで第3章を書いてください」
✅ 良い例: 「WI-00_common.md と WI-01_ch3_latter.md を読んで作業してください。他のファイルは読まないでください」
```

### 2. 作業指示書の構成

作業指示書は `docs/detailed_design/work_instructions/` に配置する。

- **WI-00_common.md**: 全設計者共通のルール（IPO形式、Mermaid図、更新履歴等）
- **WI-{番号}_{内容}.md**: タスク固有の指示。必要な情報（クラス定義、Protocol、計算式等）を直接埋め込む
- **WI-R{番号}_{内容}.md**: レビュー指示書。チェック観点・報告フォーマットを定義

### 3. 指示書に含めるべき内容

- **対象ファイルパス**: 作成/追記するファイルの絶対パス
- **必要な情報の直接埋め込み**: Protocol定義、属性一覧、メソッドシグネチャ、計算式等をコピペで埋め込む
- **成果物のフォーマット**: 冒頭テンプレート（更新履歴等）、セクション構成、記述形式（IPO形式等）
- **「他のファイルは読まない」の明示**: 必ず記載する

### 4. サブエージェントの起動ルール

- **最大並列数: 3エージェント**（レートリミット対策）
- **1エージェント1タスク**: 適切な分割粒度で分割する（下記「分割粒度の目安」参照）
- **読むファイルは最大2つ**: WI-00 + WI-{番号} のみ。追記タスクの場合は対象ファイルのReadも加わり最大3つ
- **プロンプトの定型文**:
  ```
  gridflow詳細設計者として作業してください。
  以下の2ファイルだけ読んで、指示に従ってファイルを作成してください。
  **他のファイルは一切読まないでください。**
  1. `/home/user/gridflow/docs/detailed_design/work_instructions/WI-00_common.md`
  2. `/home/user/gridflow/docs/detailed_design/work_instructions/WI-{番号}.md`
  ```

### 5. 分割粒度の目安

タスクが大きすぎるとタイムアウトやレートリミットで失敗する。以下を目安に分割する。

| 指標 | 目安 | 根拠 |
|---|---|---|
| セクション数 | 3〜5セクション/タスク | 5超でタイムアウトリスク増 |
| プロンプトに埋め込む情報量 | 2000文字以内 | WI-00(15行)+WI本体で合計2000文字が目安 |
| 生成される成果物の想定行数 | 100〜300行/タスク | 300行超は分割を検討 |
| 入力に必要なクラス/Protocol定義数 | 5定義以内 | 定義が多いと生成品質が落ちる |

**分割の判断フロー:**
1. セクション数が5以下 → 分割不要
2. セクション数が6以上 → 前半/後半に分割
3. 1セクション内のクラス数が8以上 → そのセクションだけで1タスク
4. 失敗（タイムアウト/レートリミット）した → さらに細かく分割して再試行

**実績値（本プロジェクト）:**
- 成功: WI-04(第5章, 4セクション) → 13Kトークン, 43秒, 176行
- 成功: WI-05(第7章, 6セクション) → 26Kトークン, 213秒, 655行
- 失敗: 第3章全体(9セクション) → タイムアウト
- 成功: 第3章3.1-3.4(4セクション) → 23Kトークン, 296秒, 519行

### 6. レビュアーの起動ルール

- **レビュー指示書（WI-R{番号}）を必ず先に読ませる**
- **チェック観点をチェックリスト形式で定義**: A(構造品質), B(設計原則), C(整合性), D(完成度)
- **報告フォーマットを指定**: ✅/⚠️/❌ 形式。問題なし項目はまとめて列挙、問題ありのみ個別報告
- **相互整合性レビュー（WI-R2）**: 章間ペアを明示し、チェック対象を限定する

### 7. 追記タスクの注意

既存ファイルに追記する場合:
- 「Readで既存内容を読み、末尾に追記した全体をWriteする」と明示
- 更新履歴に追記版を追加するよう指示

### 8. コミット・プッシュ

- サブエージェントの完了通知が来たらすぐにファイルサイズ確認 → コミット → プッシュ
- コミットメッセージに章番号・セクション範囲・主要内容を含める
- レビュー指摘のERROR修正は即座に対応し、別コミットで記録

## プロジェクト構成

```
docs/
├── architecture/          # アーキテクチャドキュメント（ACDM）
├── basic_design/          # 基本設計書（IPA準拠）
├── detailed_design/       # 詳細設計書（IPA準拠）
│   ├── work_instructions/ # 作業指示書（WI-00〜WI-08, WI-R1〜R2）
│   ├── README.md
│   ├── 01_requirements.md〜11_build_deploy.md
│   └── appendix.md
├── gridtwin_lab_plan.md   # 計画書
├── basic_design_plan.md   # 基本設計書作成計画
└── detailed_design_plan.md # 詳細設計書作成計画
```

## ID体系

- **REQ-B/F/Q/C-xxx**: 基本設計書の要件ID（変更不可）
- **DD-MOD/CLS/SEQ/STT/DAT/ALG/ERR/CFG/TST/BLD-xxx**: 詳細設計ID
- **E-{レイヤー2桁}{連番3桁}**: エラーコード（E-10xxx〜E-40xxx）
- **FN-xxx**: 機能ID（基本設計書定義、変更不可）

## 本ルールの更新規則

### 更新のタイミング
- サブエージェントの運用で**新しいパターンの成功/失敗**が確認されたとき
- ユーザからルール変更の指示があったとき
- プロジェクト構成やID体系に変更が入ったとき

### 更新の手順
1. 変更内容を CLAUDE.md に反映
2. 変更理由を git commit メッセージに記載（「ルール追加: 〜」「ルール修正: 〜」形式）
3. 実績値（成功/失敗のトークン数・所要時間・行数）があれば「分割粒度の目安」の実績値テーブルに追記

### 更新してはいけないもの
- ユーザの明示的な承認なくルールの**削除**や**緩和**をしない
- 特に「最大並列数」「読むファイル数の上限」は勝手に増やさない
